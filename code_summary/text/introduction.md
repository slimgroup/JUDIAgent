# Introduction

## 1.1 Scientific Context

Seismic imaging reconstructs subsurface properties—primarily acoustic velocity and density—from recordings of reflected and refracted waves generated by controlled sources. The forward problem is governed by the acoustic or elastic wave equation, a second-order partial differential equation (PDE) whose numerical solution requires careful discretization, boundary treatment, and computational resource allocation. The inverse problem, whether formulated as reverse-time migration (RTM), least-squares RTM (LSRTM), or full-waveform inversion (FWI), demands repeated PDE solves and adjoint computations to minimize data misfit functionals subject to physics constraints [1, 2].

A critical yet underexplored aspect of seismic inversion is **experimental design**: the selection of acquisition parameters—source positions, receiver arrays, recording times, sampling rates, and wavelet characteristics—that maximize the information content of recorded data while respecting operational constraints (cost, access, environmental impact). Poorly designed surveys produce data with limited subsurface illumination, leading to ill-conditioned inverse problems and artifacts in reconstructed images [3].

## 1.2 The Experimental Design Challenge

Classical experimental design for inverse problems frames the problem as optimizing an information-theoretic criterion (e.g., D-optimality, A-optimality) or a proxy objective over the space of acquisition configurations [4, 5]. However, seismic acquisition design exhibits several challenges:

1. **High dimensionality:** A typical 3D marine survey involves thousands of source-receiver pairs, each parameterized by spatial coordinates and timing variables.

2. **Non-convexity:** The mapping from acquisition geometry to image quality is highly nonlinear and non-convex, with many local optima corresponding to qualitatively different survey designs.

3. **Computational expense:** Evaluating a single design requires solving the wave equation—a costly operation even on modern GPU clusters.

4. **Expert knowledge bottleneck:** Effective survey design integrates geophysical intuition (subsurface illumination patterns), operational constraints (vessel navigation, cable management), and numerical considerations (CFL stability, memory limits).

## 1.3 Why an Agent Formulation?

We argue that the complexity of seismic experimental design calls for an **agentic** approach—a system that reasons about objectives, plans multi-step workflows, retrieves domain knowledge, and adapts based on feedback. This paradigm differs from classical optimization in several key respects:

| Aspect | Classical Optimization | Agent-Based Design |
|--------|------------------------|-------------------|
| Objective | Fixed mathematical criterion | Interpreted from natural language goals |
| Search strategy | Gradient descent / metaheuristics | LLM-guided reasoning with tool use |
| Domain knowledge | Hard-coded constraints | Retrieved dynamically via RAG |
| Feedback loop | Loss function gradient | Runtime diagnostics + human-in-the-loop |
| Generalization | Problem-specific tuning | Transfer across related tasks |

Large language models (LLMs) have demonstrated remarkable capabilities in code generation, reasoning, and tool use [6, 7]. By coupling an LLM with physics-based solvers, we obtain a system that can:

- Translate high-level objectives ("optimize source positions for deep imaging") into concrete code.
- Retrieve and adapt relevant examples from documentation and prior workflows.
- Validate generated code through linting and execution.
- Iterate based on error messages and simulation diagnostics.

## 1.4 Contributions

This work makes the following contributions:

1. **Agentic formulation of seismic experimental design:** We formalize the problem as a Markov decision process where the agent's actions correspond to code generation, parameter selection, and workflow orchestration.

2. **Integration with JUDI:** We demonstrate how LLM agents can interface with a production-quality wave-equation solver (JUDI) through a retrieval-augmented code generation pipeline.

3. **Evaluator-optimizer architecture:** We present a two-stage agent design where generated code is validated through static analysis and execution before being refined iteratively.

4. **Reproducible framework:** We release JUDIAgent as an open-source Python/Julia package with CLI, API, and IDE integration (MCP for VSCode).
